# Workflow 4: Practice Generator

## Purpose

Generate personalized and hands-on practical activities to apply knowledge from a specific subtopic of a roadmap. Activities can be coding projects, theoretical exercises, case studies, or a mix, depending on the topic and user context.

**IMPORTANT: Language Adaptation**
All questions, activities, and responses generated by this workflow must be in the same language the user writes in. If the user writes in Spanish, generate activities in Spanish. If they write in English, generate in English. If they use another language, match that language.

## When to Activate

This workflow activates when:
- User asks for "practice", "exercise", "activity", "project to apply"
- Phrases like: "generate a practical activity about [topic]", "what can I do to practice"
- User has a roadmap and wants a specific activity for a subtopic

## Prerequisites

- An existing roadmap (generated by Workflow 1 or any learning plan file)
- Path to the roadmap file
- Identification of the specific subtopic/concept to generate practice for

## Output

- Personalized practical activity in markdown format
- Includes: objective, context, detailed steps, success criteria, resources
- Format: markdown file `<subtopic>_practice_activity.md` or displayed in terminal

## PHASE 1: Identify Roadmap and Subtopic

### Step 1.1: Request Roadmap Path

If user has NOT provided the roadmap path:

**Question:** (in user's language)
"To generate a personalized practical activity, I need the path to your learning roadmap. Where is it located?"

**Expected response examples:**
- `~/learning/AI_Agents_learning_plan.md`
- `~/projects/Kubernetes_learning_plan.md`
- `./React_Frontend_learning_plan.md`

### Step 1.2: Read the Roadmap

Once you have the path:

```
Use: file_read to read the complete roadmap content
```

**If the file doesn't exist or can't be read:**
- Inform user clearly
- Ask for an alternative path or verify the file exists
- Offer alternative: "Would you like to generate an activity without a roadmap? (less personalized)"

### Step 1.3: Identify the Subtopic

If user has NOT specified which subtopic/concept to practice:

## PHASE 2: Analyze Subtopic Context

Once the subtopic is identified, extract from the roadmap:

### 2.1: Roadmap Information

**Extract:**

1. **Concept description** in the roadmap
2. **Level** (Beginner, Intermediate, Advanced)
3. **Suggested practical project** (if exists in roadmap)
4. **Prerequisites** (previous concepts needed)
5. **User context:**
   - Application scope (from roadmap profile)
   - Context (work, personal, academic)
   - Final objective (from roadmap)
   - Technological constraints (if exist)
   - Learning style (practical, theoretical, mixed)

### 2.2: Determine Activity Complexity

Based on the concept's level:

**Level 1 (Beginner):**
- Simple, step-by-step activity
- Estimated time: 1-3 hours
- Focus on applying basics
- No complex dependencies

**Level 2 (Intermediate):**
- More complex activity
- Estimated time: 3-8 hours or 1-2 days
- Focus on applying multiple concepts
- May require tool integration

**Level 3 (Advanced):**
- Integrating project
- Estimated time: several days or weeks
- Focus on solving real problem
- Integrates multiple roadmap concepts

## PHASE 3: Determine Activity Type

Based on:
- **Nature of the concept** (technical, theoretical, architectural)
- **User's learning style** (from roadmap)
- **Application context** (from roadmap)

### Available Activity Types

#### 1. **Code Project**

**When to use:**
- Concept is technical/implementation
- User has "practical" or "mixed" style
- Concept can be implemented in code

**Characteristics:**
- Objective: Implement specific functionality
- Includes: Setup, starter code (optional), tests
- Result: Functional application or component

**Example:** "Implement a basic RAG system with local vector store"

---

#### 2. **Theoretical Exercise/Analysis**

**When to use:**
- Concept is architectural or strategic
- User has "theoretical" style
- Concept requires deep understanding before implementing

**Characteristics:**
- Objective: Analyze, design, document
- Includes: Scenario, guiding questions, analysis criteria
- Result: Analysis or design document

**Example:** "Design architecture of a multi-agent system for problem X"

---

#### 3. **Case Study**

**When to use:**
- Concept is better understood with real examples
- User needs to see application in context
- Concept is complex with multiple variations

**Characteristics:**
- Objective: Analyze real case and propose improvements
- Includes: Case description, problem, data
- Result: Critical analysis and solution proposal

**Example:** "Analyze RAG implementation in ChatGPT and propose improvements"

---

#### 4. **Mix (Theory + Practice)**

**When to use:**
- Concept requires both: design and implementation
- User has "mixed" style
- Concept is sufficiently complex

**Characteristics:**
- Objective: Design first, then implement
- Includes: Design phase + implementation phase
- Result: Design document + functional code

**Example:** "Design and implement an agent that uses persistent memory"

---

### Type Decision

**Decision Matrix:**

| Concept \ Style | Practical | Theoretical | Mixed |
|-------------------|----------|---------|-------|
| **Technical/Code** | Code project | Analysis + simple code | Mix (design + code) |
| **Architectural** | Code with decisions | Theoretical exercise | Case study + prototype |
| **Conceptual** | Practical examples | Theoretical exercise | Case study |

## PHASE 5: Deliver Activity

**USE `AskUserQuestion` IF AVAILABLE**

**Question:**
"How would you like to receive the practical activity?"

**Options:**
- Show in terminal (formatted text)
- Save to markdown file (`<subtopic>_practice_activity.md`)

### Step 5.2: Deliver According to Preference

#### Option A: Terminal Only
Show the complete formatted activity in terminal.

#### Option B: File Only

1. **Naming:** `<subtopic>_practice_activity.md`
   - Example: `RAG_practice_activity.md`
   - Example: `Multi_Agent_Architectures_practice_activity.md`

2. **Location:**
   - First option: Current directory
   - If fails: Ask user where to save it

3. **Confirm:** "I've generated the practical activity and saved it at [path]"

#### Option C: Both
1. Show in terminal
2. Save to file
3. Confirm: "Here's your practical activity. I've also saved it at [path]"

### Step 5.3: Offer Next Steps using `AskUserQuestion`

After delivering the activity:

**MANDATORY: USE `AskUserQuestion`**

**Question (via AskUserQuestion):**
"Excellent! Your practical activity is ready. What's your next step?"

**Options:**
1. **Start the activity now**
   - Description: Begin working on the activity. I can help you if you have questions during the process
2. **Deep dive into a concept**
   - Description: Use Deep Dive to thoroughly explore a concept relevant to the activity (Workflow 2)
3. **Generate another activity**
   - Description: Create a practical activity for a different concept from the roadmap (repeat Workflow 4)
4. **Validate understanding with Feynman**
   - Description: Explain the concepts related to this activity and receive feedback (Workflow 3)
5. **Finish for now**
   - Description: End here and work on the activity at your own pace

### Step 5.4: Act According to Response

**If responds Option 1 (Start now):**
- Say: "Excellent, let's get started. Remember you can stop at any time if you need clarifications."
- Say: "When you have questions about any step or concept, let me know and we'll explore it in depth."
- Say: "Is there anything specific about the activity you'd like me to clarify before we start?"

**If responds Option 2 (Deep dive):**
- Say: "Great, let's deep dive into a concept that will help you with the activity."
- Say: "Is there a specific concept from the activity you'd like to explore thoroughly before starting?"
- Activate Workflow 2 (Deep Dive Learning)

**If responds Option 3 (Another activity):**
- Say: "Perfect, let's generate another practical activity for a different concept."
- Say: "For which of the roadmap concepts would you like to generate an activity?"
- Return to PHASE 1 (request roadmap path if not available or use the one already loaded)
- Repeat the Practice Generator cycle

**If responds Option 4 (Validate with Feynman):**
- Say: "Good idea, let's validate your understanding of the concepts related to this activity."
- Say: "Which concept from the activity would you like to explain and have me review with Feynman?"
- Activate Workflow 3 (Feynman Review)

**If responds Option 5 (Finish):**
- Say goodbye to user positively
- Say: "Perfect, your activity is ready. When you start working on it, I'll be here to help if you need it. Good luck with your practice!"

## Quality Guidelines

### Characteristics of Effective Activities

‚úÖ **Good activities:**
- **Clear and measurable objectives:** "At the end, you'll have a functional RAG system that answers questions about your own documents"
- **Specific and actionable steps:** "Step 1: Install ChromaDB. Step 2: Load documents in 500-token chunks"
- **Contextualized to user:** If the roadmap mentions Python, use Python; if it mentions finance, use financial examples
- **Appropriate level:** Neither too easy nor too difficult for the concept level
- **Verifiable success criteria:** Concrete checklist of what must work

‚ùå **Bad activities:**
- **Vague objectives:** "Learn about RAG" (how is that measured?)
- **Generic steps:** "Implement RAG" (no guidance)
- **Decontextualized:** Activity about medicine when user works in fintech
- **Incorrect level:** PhD-level activity for a Level 1 concept
- **No clear criteria:** "Do it well" (what is "well"?)

### Context Adaptation

**Always consider from the roadmap:**
- **Application context** (work, personal, academic)
- **Specific domain** (NLP, Computer Vision, Backend, etc.)
- **Technological constraints** (open-source only, Python 3.8+, etc.)
- **Final objective** (get a job, complete a project, etc.)
- **Learning style** (practical, theoretical, mixed)

## Error Handling

### If roadmap doesn't exist
- Offer to generate activity without roadmap (less personalized)
- Suggest creating roadmap first (Workflow 1)

### If subtopic isn't clearly in the roadmap
- Ask user if they want an activity anyway
- Offer closest subtopic from roadmap
- Generate generic activity if user confirms

### If concept is too advanced
- Warn user about the level
- Suggest completing prerequisites first
- Offer simplified version of the activity

### If file cannot be saved
- Show in terminal as alternative
- Ask for alternative location
- Offer to copy content manually
